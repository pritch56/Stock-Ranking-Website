Trading Bot League Platform

Technical Architecture & System Design Document

1. Overview
1.1 Purpose

The platform allows algorithmic trading bot developers to:

Create accounts

Connect their trading bots to the platform

Submit trades programmatically

Compete in performance leagues

Compare long-term performance metrics

The system acts as a central execution and verification layer so bots cannot falsify results.

7. Performance Calculation Engine

Runs via worker.

Metrics computed:

Return
Return = (Final Equity - Initial Equity) / Initial Equity
Sharpe Ratio
Sharpe = (Mean Return - Risk Free Rate) / Std Dev
Max Drawdown

Largest peak-to-trough drop.

Win Rate
Wins / Total Trades

10. Security Architecture

This is critical.

API Authentication

Every bot has:

bot_id
api_key

Requests validated via HMAC.

Rate Limiting

Prevent spam.

Use:

Redis

IP throttling

Example:

100 requests / minute
SQL Injection Protection

Handled via:

SQLAlchemy ORM

Parameterized queries

DDoS Protection

Use:

Cloudflare

Features:

IP filtering

WAF

bot detection

Encryption

All traffic via:

HTTPS
TLS 1.3
Secret Storage

Secrets stored using:

Supabase vault

environment variables

Never commit keys to git.

11. Anti-Cheat System

Prevent bots gaming rankings.

Methods:

Execution verification

Platform executes trades itself.

Bots cannot fake fills.

Slippage model

Simulated trading includes:

price + slippage
Position limits

Prevent unrealistic leverage.

Example:

max_position = 100% equity
Trade frequency checks

HFT bots separated into league.
